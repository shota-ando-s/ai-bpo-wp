よくある失敗は「マニュアルを作らずに引き継ぎ、担当者の頭の中にある知識だけで業務を渡そうとする」ことです。これをやると、移行後に品質が急落し、ベンダーからの質問が止まらなくなります。

## 移行準備フェーズでやること

移行準備（Step 6）では以下の4つを行います。

1. **業務マニュアルの作成**（属人的な知識の文書化）
2. **引き継ぎスケジュールの策定**
3. **並行稼働期間の設計**
4. **データ・システムアクセスの整備**

---

## ①業務マニュアルの作成

### なぜマニュアルが重要か

BPOベンダーのオペレーターは、自社の業務知識をゼロから学びます。「察してほしい」「常識でわかるはず」は通用しません。マニュアルに書かれていないことはできないと思ってください。

また、マニュアルを作る過程で「なぜこの手順でやっているのか誰も知らない」「ルールが担当者によって違う」という問題が浮き彫りになります。移行前にこれを解消することが、移行後の品質向上につながります。

### マニュアルに含める5要素

| 要素 | 内容 |
|---|---|
| **目的・概要** | この業務は何のためにあるか |
| **前提条件・必要情報** | 作業前に何が揃っている必要があるか |
| **手順（ステップ）** | 何をどの順番でやるか（スクリーンショット付き） |
| **判断基準・例外処理** | 「こういう場合はどうする」の一覧 |
| **エラー・質問時の連絡先** | 不明点が出た際の窓口 |

### マニュアル作成の具体的な手順

**ステップ1：業務の録画・記録（30分〜）**

担当者に実際の作業をしてもらいながら、画面録画（Loom、Zoom等）で記録します。同時に音声で「今何をしているか」を説明してもらいます。スクリーンショットの手動撮影は非効率なため、画面録画が最も効率的です。

**ステップ2：録画をもとに文字起こし**

録画内容をLLMに処理させてマニュアルの骨格を作ります（後述のAI活用法参照）。

**ステップ3：判断基準・例外処理の追記（最重要）**

手順書に落ちやすいのが「例外処理」です。「請求書の金額が前月と30%以上乖離している場合は担当者に確認する」「○○会社の請求書は必ずPDF原本を保存する」など、過去のトラブルや特殊ルールをヒアリングして追記します。

**ステップ4：ベンダーによるリハーサルとフィードバック**

完成したマニュアルをベンダーに渡し、実際に作業してもらいます。「マニュアル通りにやれなかった箇所」「分からなかった箇所」を全てフィードバックしてもらい、マニュアルを修正します。この工程を最低2回繰り返してください。

## AIを使ってマニュアル作成を1/3の時間にする

### 手順1：業務動画→テキスト変換

LoomやZoomで録画した業務動画を、以下のツールで文字起こしします。

- **Whisper（OpenAI）**：高精度な音声→テキスト変換、無料で使えるAPI
- **Notion AI**：Notionに取り込んで文字起こし＋整理
- **Adobe Premiere Pro（Captions機能）**：動画に字幕付けしながら文字起こし

**Whisper APIでの文字起こし手順（Python）：**
```python
from openai import OpenAI
client = OpenAI()

with open("business_video.mp4", "rb") as f:
    transcript = client.audio.transcriptions.create(
        model="whisper-1",
        file=f
    )
print(transcript.text)
```

### 手順2：文字起こしをLLMでマニュアル化

文字起こしテキストをLLMに入力し、業務マニュアルの形式に変換します。

**プロンプト例：**
```
以下は業務担当者が作業中に録音した音声の文字起こしです。
これをもとに、BPOオペレーター向けの業務マニュアルを作成してください。

【マニュアルの構成】
1. 業務概要（目的・担当者・頻度）
2. 必要な情報・ツール（前提条件）
3. 作業手順（ステップ形式、各ステップに概要と説明）
4. 判断基準・例外処理の一覧
5. よくある質問と回答
6. 不明点・エラー時の連絡先

【注意点】
- 初めてこの業務を行う人でも理解できる表現で書く
- 専門用語には括弧書きで説明を加える
- 曖昧な表現（「適切に」「できるだけ早く」）は避ける

---
【文字起こしテキスト】
[文字起こし内容を貼り付け]
```

このプロンプト1回で、業務マニュアルの70〜80%が完成します。残りの20〜30%は判断基準・例外処理・連絡先等の社固有情報を人間が追記します。

### 手順3：スクリーンショットの自動挿入

業務動画から特定のフレームを自動抽出してスクリーンショットとしてマニュアルに挿入するツールも存在します。Loomの場合は、自動スクリーンショット機能が内蔵されています。

## ②引き継ぎスケジュールの策定

### 標準的な引き継ぎスケジュール

```
Week 1〜2：マニュアル作成（社内作業）
Week 3   ：マニュアルのベンダー確認・フィードバック
Week 4   ：ベンダートレーニング（マニュアルを使った研修）
Week 5〜6：並行稼働（社内とベンダーが同じ業務を処理）
Week 7〜8：ベンダー主導に移行（社内はチェック・フォロー）
Week 9〜 ：Step 7（パイロット運用）へ移行
```

業務の複雑度によって、この期間は2〜3倍になることもあります。「複雑な業務ほど移行準備に時間をかける」が鉄則です。

### 引き継ぎスケジュールに含める項目

| 日程 | 内容 | 担当 |
|---|---|---|
| W1〜W2 | マニュアル作成・レビュー | 社内担当者 |
| W3 | ベンダーへのマニュアル配布・確認依頼 | 社内PJ担当 |
| W3〜W4 | ベンダートレーニング実施 | 社内・ベンダー合同 |
| W5 | 並行稼働開始（社内：全件、ベンダー：同じ業務を同時処理） | 両者 |
| W6 | 並行稼働の結果比較・差異分析 | 社内QA担当 |
| W7〜W8 | ベンダー主導（社内は抜き取りチェック） | ベンダー主 |

## ③並行稼働期間の設計

並行稼働とは、移行期間中に「社内でもベンダーでも同じ業務を同時に処理する」ことです。

### 並行稼働の目的

- ベンダーの処理結果と社内の処理結果を比較し、差異を特定する
- マニュアルの不備・誤りを発見する
- ベンダーのスキルを段階的に上げながらリスクを最小化する

### 並行稼働の設計ポイント

**比率の設定：**
```
Week 5：社内100%、ベンダー20%（同じ業務を処理して比較）
Week 6：社内80%、ベンダー50%（比較しながら習熟度確認）
Week 7：社内20%（チェックのみ）、ベンダー100%
```

**差異発見時の対応フロー：**
差異（ベンダーの処理ミス）が発生した場合：
1. 原因分類：マニュアルの不備？ベンダーの理解不足？ 例外パターンの未記載？
2. 原因に応じてマニュアル修正またはベンダーへの追加トレーニング
3. 修正後に再度並行稼働で確認

## ④データ・システムアクセスの整備

ベンダーが業務を遂行するために必要なアクセス権を整備します。

### チェックリスト

```
□ 使用システムへのアカウント発行（最小権限の原則で）
□ 共有ファイルストレージの設定（Google Drive、SharePoint等）
□ コミュニケーションツールへの招待（Slack、Teams等）
□ アクセスログの取得設定
□ 退職・契約終了時のアクセス失効手順の確立
```

**セキュリティ注意事項：**
ベンダーオペレーターには「必要最小限のアクセス権」のみ付与してください。「とりあえず全部アクセスできるようにしておく」はセキュリティリスクです。業務が終了したら速やかに権限を削除する仕組みも作っておきましょう。

## 移行準備でよくある失敗パターン

### 失敗1：担当者任せで作ったマニュアルが第三者に伝わらない

担当者が自分目線でマニュアルを書くと、「この意味がわからない」「この手順が足りない」という問題が起きます。**必ず「この業務を全く知らない人」がマニュアルを見て実際にできるかをテストしてください。**

### 失敗2：並行稼働を省略する

「早く始めたい」という気持ちから並行稼働を省いて直接ベンダー主導に切り替えると、品質問題が本番運用で露見します。並行稼働は保険料だと思ってください。

### 失敗3：担当者が「引き継ぎに非協力的」になる

「自分の仕事を取られる」と感じた担当者は、マニュアル作成をサボったり、故意に情報を出し惜しみしたりします。「空いた時間でより重要な業務に集中してもらう」という具体的な次の役割を提示することが必須です。

### 失敗4：移行後の「社内の窓口」が明確でない

ベンダーが質問したい時に「誰に聞けばいいか」が分からない状態は最悪です。必ず「業務担当の社内窓口（メイン・サブ）」を決め、応答時間のルール（例：「Slackの質問には4時間以内に回答」）も設定してください。

## よくある質問（FAQ）

**Q. 業務マニュアルはどれくらいの分量が必要ですか？**
業務1つにつきA4換算で5〜15ページが目安です。単純な業務は短く、複雑な業務（判断基準・例外が多い）は長くなります。分量より「初めての人でも作業できるか」を判断基準にしてください。

**Q. LLMが作ったマニュアルの精度はどれくらいですか？**
文字起こしを元にLLMが生成したマニュアルは、手順の骨格（70〜80%）は正確ですが、判断基準・例外処理・社内特有のルール（20〜30%）は担当者の追記が必要です。「完全自動生成」ではなく「半自動化ツール」として使うのが正しい認識です。

**Q. ベンダーがトレーニング費用を請求してきた場合はどうすればいいですか？**
一般的に「初期トレーニング費用」はベンダー負担とすることが多いです。ただし、業務が非常に複雑・特殊な場合は委託者負担になることもあります。Step 3（RFP）の段階でトレーニング費用の取り扱いを明記しておくことが重要です。

**Q. 引き継ぎ中に担当者が退職してしまった場合はどうすればいいですか？**
これがBPO導入の最大リスクシナリオのひとつです。移行準備開始と同時に「担当者がいなくても業務が回る状態」を目指してマニュアルを整備してください。また、複数の担当者に業務知識を分散させ（ナレッジの属人化解消）、早期退職リスクに備えることをお勧めします。

**Q. システムのアクセス権付与にITの専門知識が必要ですか？**
クラウドサービス（Google Workspace、Microsoft 365等）であれば管理者が操作できます。オンプレミスのシステムや社内開発システムの場合はIT担当者の関与が必要です。この作業を見落として移行直前に発覚するケースが多いため、Step 2の段階からIT担当者を巻き込んでおいてください。

**Q. マニュアルの更新はどう管理しますか？**
業務プロセスの変更・法改正・システム更新等があった場合にマニュアルを更新するルールを設けてください。更新履歴（いつ・誰が・何を変更したか）を残すことと、最新版をベンダーに通知する仕組みが必要です。GoogleドキュメントやNotionで管理すると、更新通知が自動化できます。

## まとめ

移行準備（Step 6）で行うことは4つです：

1. **業務マニュアルを5要素で作成する**（目的・前提・手順・判断基準・連絡先）
2. **LLMと動画文字起こしで作成時間を1/3に短縮する**
3. **並行稼働期間を必ず設けて品質を段階的に確認する**
4. **データ・システムアクセスを最小権限で整備する**
