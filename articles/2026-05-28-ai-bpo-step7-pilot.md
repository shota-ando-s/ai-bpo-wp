[Step 6で移行準備が完了](/ai-bpo-step6-migration)し、ベンダーへのトレーニングも終わりました。いきなり本格稼働に移るのではなく、まずは「パイロット運用（試験運用）」を実施します。

パイロット運用はBPOプロジェクト最後の安全網です。ここで問題を発見・修正してから本格稼働に移ることで、本番でのトラブルを大幅に減らせます。「早く本格稼働したい」という気持ちはわかりますが、パイロットを省略することは絶対に避けてください。

## パイロット運用の目的

パイロット運用で確認することは5つです：

1. **品質（エラー率）**：SLAの数値を達成できるか
2. **スピード（処理速度）**：納期を守れるか
3. **コミュニケーション**：質問・不明点の対応が適切か
4. **マニュアルの充足度**：マニュアル通りにやれない箇所はどこか
5. **例外処理の対応力**：想定外の状況への対応は適切か

## パイロット運用の設計

### 対象範囲の設定

全業務量の20〜30%をパイロット対象にすることをお勧めします。

**対象の選び方：**
- 最もボリュームが多い「典型的な業務」を中心に選ぶ
- ただし「最も難しい業務」も最低1〜2件含める（実力テスト）
- 個人情報・機密情報に関わる業務は最後に移行（パイロット後半で検証）

**具体例：**
月間200件の請求書処理をBPOする場合：
- パイロット対象：月50件（25%）
- 残り150件：社内で引き続き処理
- パイロット期間中は社内でもチェック（品質比較）

### パイロット期間の設定

**最低4週間、推奨8週間**が目安です。

短すぎる理由でパイロットを短縮することは避けてください。特に以下の理由で8週間が推奨です：

- 最初の2週間はベンダーが慣れておらずエラーが多い（学習曲線）
- 月末・月初などの業務サイクルを最低1周経験させる
- 繁忙期・閑散期の両方をカバーする

### KPIの事前設定（最重要）

パイロット開始前にKPI（評価指標）と判定基準を決めておきます。「終わってから評価する」と、人によって評価がばらつき、本格稼働への移行判断で揉めます。

**標準的なパイロットKPI：**

| KPI | 基準値（SLAより少し緩め） | 測定方法 |
|---|---|---|
| エラー率 | 1.0%以下（本番SLAは0.5%以下） | 月次レポート |
| 処理速度 | 72時間以内（本番SLAは48時間以内） | タイムスタンプ記録 |
| 質問件数 | 週10件以下（慣れるにつれ減少を確認） | Slack/メールログ |
| KPI達成率 | 月4週のうち3週達成 | 週次集計 |

**本格稼働移行の判断基準（事前に合意する）：**
```
以下の全条件を満たした場合、本格稼働に移行する：
□ 8週間のパイロット期間を完了した
□ 最後の4週間でエラー率が基準値（1.0%）以下
□ 最後の4週間で処理速度が基準値（72時間）以内
□ 週次の質問件数が5件以下に減少している
```

この判断基準を事前にベンダーと書面（または契約書の付属書）で合意しておくことが重要です。

## パイロット中のモニタリング体制

### 社内モニタリング体制

**週次レビューの実施：**

| 頻度 | 内容 | 参加者 |
|---|---|---|
| 毎日 | ベンダーの処理状況確認（件数・エラー） | PJ担当者 |
| 週次 | KPI集計・問題共有・改善策合意 | PJ担当・業務担当・ベンダー |
| 月次 | SLA達成状況・総合評価 | 経営者（または管理職）・ベンダー責任者 |

### AIモニタリングで異常を早期発見する

手動でエラーを確認するだけでは、問題を発見したときには影響が広がっている可能性があります。AIモニタリングを導入することで、異常を発生時点でリアルタイムに検知できます。

**AIモニタリングの仕組み：**

1. ベンダーの処理結果データをリアルタイムで受け取る
2. AIが「通常範囲」から外れた処理を自動検知
3. 異常を担当者にアラート通知

**具体的な異常検知のルール例：**

| 異常パターン | 検知条件 | アクション |
|---|---|---|
| エラー率の急増 | 1日のエラー率が基準値の3倍超 | Slackアラート → 即時確認 |
| 処理速度の遅延 | 受取から36時間経過した未処理件数が5件超 | 担当者へ通知 |
| 処理量の急減 | 前日比50%以下の処理件数 | ベンダーへ稼働確認 |
| 深夜処理 | 22時〜翌7時の処理ログ | セキュリティ確認トリガー |

**ツールの選択：**

- **ローコードで実現**：Zapier、Make（旧Integromat）でルールベースの自動アラートを設定
- **Excelベース**：条件付き書式＋Power Automateでメール通知
- **クラウド監視**：Datadog、Google Cloud Monitoringなど（データがAPI連携できる場合）

## フィードバックサイクルの回し方

パイロット中に発見した問題は、以下のサイクルで素早く修正します。

```
問題発見 → 原因分類 → 対策決定 → マニュアル修正 → ベンダーへ通知 → 確認
  （1〜2日）   （1日）    （1日）       （1日）         （即日）   （翌週）
```

**原因の分類（最重要）：**

問題が起きたときに「ベンダーのせい」と決めつける前に、以下の3つの原因を確認してください：

| 原因 | 対策 |
|---|---|
| **マニュアルの不備**（手順が曖昧・例外が未記載） | マニュアル修正→ベンダーに再配布 |
| **ベンダーの理解不足**（マニュアルは正しいが実行できていない） | 追加トレーニング・ロールプレイ実施 |
| **業務の複雑さの問題**（想定より複雑で委託が困難） | スコープを見直す・自動化ツール導入を検討 |

## パイロット運用でよくある問題と対処法

### 問題1：エラー率が基準を超え続ける

**確認すべきこと：**
- マニュアルに記載されていない例外が多発していないか？
- ベンダーの担当者が習熟するのに時間がかかっているだけか（学習曲線の問題）？
- ベンダーのスキルそのものが不十分か？

**対処：** 原因を特定してから対策を打つ。原因不明のまま「もう少し様子を見る」は禁物です。

### 問題2：ベンダーからの質問が減らない

週を追うごとに質問が減らないのは、マニュアルが不完全なサインです。質問の内容を分類し（「例外処理の質問」「手順の確認」等）、最も多いカテゴリのマニュアルを重点的に改善してください。

### 問題3：社内担当者がチェックに時間を取られすぎる

パイロット期間中は社内のチェック工数が増えます。これが想定以上に大きい場合、抜き取り検査の比率を段階的に下げることで負荷を軽減できます。ただし、抜き取り率を下げすぎると問題を見逃すリスクがあるため、AIモニタリングで補完することをお勧めします。

### 問題4：ベンダーがパイロット結果を有利に見せようとする

まれに、ベンダーがパイロット期間中だけ特別な体制を組んで成果を良く見せることがあります。これを防ぐには、KPIの測定を社内主導で行う（ベンダーの自己報告だけに頼らない）ことが重要です。

## 本格稼働への移行判断

パイロット期間終了後、以下のプロセスで移行を判断します。

**移行判断会議（ゴーノーゴー会議）：**
参加者：PJ担当者・業務担当者・経営者（管理職）・ベンダー責任者

判断のフレームワーク：
```
【GO（本格稼働移行）条件】
- 全KPIが基準値を達成している
- マニュアルの修正が完了し、ベンダーに周知されている
- 社内チェック体制が整っている

【CONDITIONAL GO（条件付き移行）条件】
- 一部KPIが未達だが、原因と対策が明確で改善中
- 本格稼働後も月次レビューを強化して監視する

【NO GO（パイロット延長または再検討）条件】
- 複数のKPIが基準値に達していない
- 原因が不明または対策の見通しが立たない
- ベンダーの体制・スキルに根本的な問題がある
```

## よくある質問（FAQ）

**Q. パイロット期間中にベンダーとの関係が悪化した場合はどうすればいいですか？**
パイロット期間は「本契約前の試用期間」として設計するのが理想です。契約書に「パイロット期間中は本格稼働義務を負わない」「パイロット終了後に継続するかを双方で合意する」と明記しておくと、関係が悪化しても撤退しやすくなります。

**Q. KPIが達成できなかった場合、ベンダーに何か請求できますか？**
パイロット期間中のペナルティは設定しないことが一般的です（ベンダー側も習熟中のため）。ただし、「本格稼働後のSLA違反」はStep 5で設定したペナルティ条項を適用します。

**Q. パイロット業務の処理コストはどちらが負担しますか？**
一般的にはパイロット期間から委託料が発生します（無料パイロットはほぼありません）。ただし「パイロット期間は本番料金の80%」などの割引交渉は可能です。

**Q. パイロット対象を増やして早く本格稼働できますか？**
業務量の50%以上をパイロット対象にすると、社内の負担（チェック工数）が大きくなりすぎます。段階的に比率を上げる方が現実的です。

**Q. ベンダーが「パイロットは不要、すぐ本格稼働で問題ない」と言った場合は？**
慎重に判断してください。実績が豊富なベンダーが自信を持って言う場合は正当性がありますが、パイロットを省略させたい（早く安定収益を確保したい）という動機の可能性もあります。少なくとも2〜4週間の試行期間は必ず設けてください。

**Q. AIモニタリングの導入には専門知識が必要ですか？**
ZapierやMakeのようなローコードツールなら、プログラミングの知識なしで実装できます。「特定の条件（エラー率X%超）が発生したらSlackに通知する」というシンプルなフローであれば、1〜2時間で設定できます。

## まとめ

パイロット運用（Step 7）で行うことは4つです：

1. **対象範囲（20〜30%）・期間（最低4週間）・KPIを事前に設定する**
2. **週次レビューで問題を素早くフィードバックし、マニュアルを改善する**
3. **AIモニタリングでリアルタイムに異常を検知する**
4. **事前に合意したKPI基準で「本格稼働移行・延長・中止」を判断する**
