> ハブ記事：[AI-BPO導入支援サービス全体像](/hikitsugi-ai-support-hub)

Step 1の業務棚卸しで「自社に何の業務があるか」が見えたら、次の問いは「その中のどれをアウトソースするか」です。

このスコープ定義は、BPO導入プロジェクトの中で**最も重要な意思決定**です。ここで範囲を広げすぎると移行リスクが高まり、狭すぎるとコスト削減効果が出ない。曖昧なままベンダー選定に進むと、後で「こんなつもりじゃなかった」というトラブルが必ず起きます。

ヒキツギAIはワークショップ形式の意思決定支援と、スコープ定義書の作成を通じてこのステップをサポートします。

---

## なぜスコープ定義が難しいか

### 問題1｜「全部やってほしい」vs「あれはダメ、これもダメ」

経営者は「コストを下げたい」という目標から「できるだけ広く外注したい」と考えがちで、現場担当者は「自分の仕事を取られる」という不安から「あれも無理、これも難しい」と言いがちです。

この両者の間で合理的な判断を下すには、**客観的な判断基準**が必要です。

### 問題2｜「段階的にやるか、一気にやるか」の設計が難しい

全業務を一度にアウトソースするのはリスクが高い。しかし少なすぎると効果も小さい。「まず何から始めて、次に何を追加するか」というロードマップ設計には経験が必要です。

### 問題3｜「決めたつもり」で進んで後でズレる

口頭で「経理をアウトソースしよう」と決めても、「請求書処理だけか、月次決算も含むのか」が書面で明確になっていないとベンダーとの交渉でズレが生じます。

## ヒキツギAIのStep 2支援 全体の流れ

### フェーズ1｜スコープ選定ワークショップ（半日）

Step 1の棚卸しレポートをベースに、経営者・管理職と一緒に「アウトソース候補」を絞り込む半日ワークショップを実施します。

**ワークショップの構成（4時間）：**

| 時間 | 内容 |
|---|---|
| 60分 | Step 1結果の確認・優先度マトリクスの読み合わせ |
| 90分 | 各業務のアウトソース可否を参加者全員でジャッジ |
| 60分 | 「第一弾スコープ」と「将来拡張候補」の分類 |
| 30分 | 合意事項の確認・次ステップの説明 |

**私たちがファシリテートする際に使う判断フレームワーク：**

各業務を4つの問いで判断します：

1. **この業務がなくなったら、顧客に直接影響が出るか？**
   → 出る場合はコア業務の可能性が高い（社内残し検討）

2. **この業務のやり方を文書化できるか？**
   → できない場合は属人化が高く、移行リスクが大きい

3. **この業務で扱う情報の機密度は？**
   → 高い場合はセキュリティ要件が厳しく、ベンダー選定に条件が増える

4. **この業務を外部が行っても、品質を客観的に測定できるか？**
   → 測定できない場合はSLA設計が難しい

---

### フェーズ2｜スコープ深掘り分析（1週間）

ワークショップで「アウトソース候補」に選ばれた業務について、より詳細な分析を行います。

**分析項目：**

**①業務の境界線定義**

「経理業務をアウトソース」ではなく、「どこまでがアウトソース範囲か」を明確にします。

例：経理業務の場合
```
【アウトソース範囲】
✅ 請求書のデータ入力・仕訳
✅ 入金確認・消込
✅ 月次試算表の作成

【社内残し】
❌ 取締役承認が必要な支払い判断
❌ 税理士との折衝
❌ 資金繰り管理
```

**②インターフェース設計（連携点の整理）**

アウトソース業務と社内業務の「境界」でのデータ連携・承認フロー・例外処理の担当を整理します。ここが曖昧だとパフォーマンス低下・エラー増加の原因になります。

**③フェーズ分け計画**

いきなり全業務をアウトソースするのではなく、リスクの低いものから段階的に移行するロードマップを設計します。

例：
- **第1フェーズ（3ヶ月後）**：請求書データ入力（月40時間）
- **第2フェーズ（6ヶ月後）**：入金消込・月次試算表作成（月30時間）
- **第3フェーズ（12ヶ月後）**：給与計算データ入力（月20時間）

---

### フェーズ3｜スコープ定義書の作成（1週間）

分析結果をもとに「スコープ定義書」を作成します。これはStep 3のRFP作成・Step 4のベンダー選定の基礎資料となります。

**スコープ定義書の構成：**

```
1. アウトソース対象業務一覧
   - 業務名
   - 具体的な作業内容（箇条書き）
   - 対象外（除外事項）の明記
   - 月間件数・工数

2. 業務境界線（責任分界点）
   - 「どこまでが委託業務か」を図で示す
   - 例外・エスカレーション時の対応フロー

3. フェーズ移行計画
   - 第1弾・第2弾・第3弾の範囲と時期

4. 前提条件・制約事項
   - 使用システム・データ形式
   - セキュリティ・コンプライアンス要件
   - 対応時間帯・言語

5. 成功指標（KPI候補）
   - 処理件数・処理精度・ターンアラウンドタイム等
```

## ヒキツギAIが特に重視するポイント

### 「社内残し業務」を明確にすることの重要性

スコープ定義書には「アウトソース範囲」だけでなく、「社内に残す業務」と「その理由」を明記することを私たちは推奨しています。

理由：
- ベンダーが「これも私たちがやります」と提案してきた際の判断基準になる
- 将来の担当者が「なぜこれをアウトソースしていないか」を理解できる
- 監査・内部統制の観点から、委託・非委託の根拠を残せる

### スコープの「広げすぎ」「狭すぎ」を防ぐ

**広げすぎのリスク：**
- 移行失敗時のインパクトが大きい
- ベンダー依存度が高まり、交渉力が下がる
- 社内のBPO管理工数が増える

**狭すぎるリスク：**
- コスト削減効果が出ない
- ベンダーにとって案件の魅力が下がり、優秀なベンダーが応募しない

私たちは「最初は小さく始めて、成功したら広げる」というアプローチを推奨します。ただし「小さすぎて意味がない」にならないよう、月削減効果が最低**20〜30万円以上**になるスコープ設計を目指します。

## よくある質問

**Q：ワークショップへの参加人数に制限はありますか？**
A：推奨は5〜8名です。多すぎると議論が収束しにくく、少なすぎると見落としが生じます。意思決定権者（経営者・部門長）が必ず含まれることが重要です。

**Q：スコープを決めた後、変更することはできますか？**
A：契約前であれば変更可能です。ただし、ベンダーとの交渉が始まった後に大きな変更が生じると、追加費用が発生することがあります。Step 2で十分な時間をかけることを推奨します。

**Q：「これはアウトソースできるか？」という個別相談はできますか？**
A：はい。ワークショップ以外にも、特定業務についてのスポットアドバイスを行います。
